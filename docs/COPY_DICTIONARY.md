# KILN Copy Dictionary (v1.0)

This is the canonical copy reference for all user-facing UI text.

Use with:
- `docs/BRANDING.md`
- `docs/Technical Requirements.md`
- `docs/EXPERIENCE.md`

---

## Approved UI-State Vocabulary

### Status and Visibility

- `VISIBLE` — published/live/in view
- `IN VIEW` — currently displayed
- `DARK` — unavailable/offline/not surfaced
- `NO SIGNAL RETURNED` — empty/404/no-result state

### Transitions and Load States

- `ACQUIRING` — loading in progress
- `RESOLVED` — loading complete
- `FIXED POSITION` — stable/static state

### Interaction Labels

- `APPROACH` — hover/approach cue
- `SURFACE` — open/read more/expand
- `MAKE CONTACT` — contact pathways

### Object Metadata

- `OBJECT 014` style object index labels
- `LAST TRANSMISSION:` as update metadata prefix

---

## Preferred Taxonomy

- `OBJECT ARCHIVE` for work index framing
- `SIGNAL` for essays/notes surfaces
- `AVAILABLE WORK` for services/collaboration route framing
- `TOOLS` for shop/store framing
- `SYSTEM` for colophon/documentation route framing

Preferred word choices:
- `Objects` over `Content`
- `Studio` over `Agency`
- `Systems` over `Solutions`
- `Archive` over `Feed`
- `Available` over `For sale`

---

## Prohibited or Retired Language

Do not use startup-space mission language in UI copy:
- launch, orbit, deploy, mission, countdown, boost

Do not use legacy machine-syntax labels:
- JCL/FORTRAN/Assembly-style labels and mnemonics

Do not use hype language:
- revolutionary, game-changing, cutting-edge
- unlock, unleash, supercharge
- frictionless, seamless
- optimize, maximize
- don't miss out, limited time

---

## CTA Style Rules

- Keep CTA language plain and specific.
- Prefer direct actions over persuasive phrasing.
- Avoid urgency and sales pressure.
- Keep metadata language in metadata slots only.

Examples:
- `SURFACE` + `View details`
- `MAKE CONTACT`
- `Return to archive`

---

## Route-Level Defaults

- `/` should read as a studio entrance.
- `/work` should read as an object archive.
- `/signal` should read as an archive of essays and notes.
- `/system` should prioritize philosophy and implementation clarity.
- `/services` should frame selective collaboration, not service-menu sales copy.
- `/shop` should frame products as tools with utility and documentation.
